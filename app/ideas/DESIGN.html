1. 定位与目标

点子库是一个「想法 × 对话 × 文档」三位一体的私人创作助手：

  • 每个 点子 是一份会持续演化的 markdown 文档（"这个点子到底是什么"的当下最佳表述）。
  • 每个点子下能开任意多场 对话，从不同角度跟 LLM 讨论它（验证用户、设计交互、追问竞品、撕方案、写文案……）。
  • 聊到有价值的内容时，一键把这场对话「整理进点子文档」，由 LLM 重写出新版文档，旧版进历史。

核心张力：聊天可以发散，文档必须收敛。 不让对话直接污染文档，而是用一次显式动作把信号沉淀下来。

跟 agent_lord 的关系： agent_lord 是「一份设计文档 + 多对话」。点子库是「多份点子文档 + 每份点子下的多对话」。 点子库 ≈ agent_lord × N，但加一个新原语：对话 → 文档的 distill 动作。 后端范式抄 app/agent_lord/api/research.py 即可，多套一层 idea 层级。

2. 核心使用流

  1. 新建点子 → 给个标题（文档暂时是空的）。
  2. 点开点子 → 看到文档（空 / 当前版本）+ 已有对话列表。
  3. 新建对话 → 跟 LLM 聊，prompt 里自动注入当前文档作为上下文。
  4. 聊到关键处 → 按「整理到文档」→ 后台 LLM 输出新版文档全文 → 覆盖写入，旧版进历史。
  5. 再开新对话 → LLM 看到的是已沉淀过的最新文档，对话起点更高。

关键体验：第一次开聊 = 跟一个完全不了解你点子的人讲。第 N 次开聊 = 跟一个已经读完最新设计稿的合伙人讲。

5. API 路由 prefix: /api/ideas

方法路径作用
GET/ideas列点子（按 updated_ts desc）
POST/ideas新建点子，body: {title?}
GET/ideas/<iid>详情：meta + doc_markdown + 对话列表摘要
PATCH/ideas/<iid>改 title 或手动改 doc，body: {title?, doc?}
DELETE/ideas/<iid>删点子（连带删全部对话和历史）
GET/ideas/<iid>/history历史版本列表 [{ts, chars}]
GET/ideas/<iid>/history/<ts>取某个历史版本的 markdown
POST/ideas/<iid>/history/<ts>/restore把历史版本还原为当前（旧"当前"也存历史）
GET/ideas/<iid>/conversations本点子下的对话列表
POST/ideas/<iid>/conversations新建对话
GET/ideas/<iid>/conversations/<cid>对话详情（消息流）
POST/ideas/<iid>/conversations/<cid>发消息（MVP 同步）；body: {message}
DELETE/ideas/<iid>/conversations/<cid>删对话
POST/ideas/<iid>/conversations/<cid>/distill核心动作：把本对话整理进文档
关于流式：MVP 先用同步路径（_call_claude，跟 agent_lord 一样），180s 超时内出结果。 体验觉得慢了再迁到壳子 SSE：把本对话桥接到 /api/conversations/<cid>/stream，prompt 由 sub-app 注入。 不要 sub-app 自己起流式 subprocess（详见 CLAUDE.md §1.2 的踩坑清单）。

6. 核心机制：对话 → 文档

这是这个产品的唯一新原语。三种可选实现路径：

方案触发优点缺点
A 显式按钮 MVP 用户在对话页点「整理到文档」 简单、可控、失败影响小 用户得记得点
B 隐式输出 主对话 prompt 里允许 LLM 在 <doc>...</doc> 块里输出新版 无操作成本 很容易误更新；流式中难裁剪展示
C 后台自动 每轮回复后台多调一次 distill 体验丝滑 成本翻倍；用户聊野时文档会乱跳

选 A 做 MVP。 跑通后视体验决定要不要叠 B/C。

Distill 调用流程

  1. 前端 POST /conversations/<cid>/distill，可选 body {scope: "all" | "recent_n", n?: 8}，默认 all。
  2. 后端读：当前文档 + 该对话全部（或最近 N 轮）消息。
  3. 拼 distill prompt（见 §8.2），同步调 _call_claude。
  4. 校验返回：非空、不是 [错误：...]、长度合理（≥10 字符 & ≤ 当前文档的 10 倍，防爆炸）。
  5. 把旧文档拷到 history/<iid>.<ts>.md，新版写入 <iid>.md。
  6. 更新 idea index：updated_ts、doc_chars；更新 conv index：distilled_count += 1。
  7. 返回 {doc: "新版 md", history_ts: N, diff: {added: 12, removed: 3}}。

失败时不动文件，返回 {error: "..."}，前端 toast 提示。

7.3 关键交互细节

  • 「整理到文档」按钮位置：对话页顶部右上 / 移动端是底部输入区上方的小条；按下后变 loading 状态，整段时间禁用输入框。
  • distill 成功后弹一个 非模态 提示条："文档已更新 · +12 / -3 行 · 查看 / 撤销"。撤销 = 调 /history/<prev_ts>/restore。
  • 对话列表里，至少 distill 过一次的对话标个 ✓ 角标，便于知道哪些对话已被沉淀。
  • 文档预览长度限制：mobile 折叠条只显示前 3 行，点开后全屏 sheet；桌面默认展开高度 60vh，超出滚动。
  • 编辑文档：MVP 用 <textarea> 直接改，PATCH 保存。富文本编辑器以后再说。

8. Prompt 设计

8.1 对话主 prompt（每轮发消息时）

你正在帮{用户名}打磨一个产品点子。下面是这个点子的当前文档和最近对话。
你的回答要简洁、具体、直接，必要时反驳他的想法。默认中文，除非他用英文。
不要客套、不要总结自己刚说过的话，不要堆砌 markdown 标题。

=== 当前点子文档 ===
{doc 全文，如果为空填 "（文档暂时为空，刚开始聊这个点子）"}
=== 文档结束 ===

此前对话（最近 12 轮）：
用户：{...}
你：{...}
...

用户：{新消息}
你：

8.2 Distill prompt（整理到文档）

你正在帮{用户名}整理一个产品点子的设计文档。下面是当前文档和最近这场对话。
请基于对话内容，输出新版文档（markdown），要求：

- 保留当前文档的核心结构和已有有效内容
- 把对话里有价值的新想法、新判断、新决定，融进对应章节
- 如果对话推翻了文档里的某个判断，更新它，不要保留矛盾的两份
- 对话里的发散、闲聊、反例可以不收录
- 不要添加"更新日志""本次修改"这类元信息
- 不要解释你做了什么，直接输出 markdown 文档全文
- 不要用 ```markdown 代码块包裹

=== 当前文档 ===
{doc 或 "（文档暂时为空）"}
=== 文档结束 ===

=== 本场对话 ===
用户：{...}
你：{...}
...
=== 对话结束 ===

10. 待定问题（请拍板或留默认）

#问题默认提案
Q1 Distill 范围：整个对话 vs 用户选择的消息？ 整个对话（最简）。后期加 "选段 distill"
Q2 点子文档要不要预置模板章节（问题/方案/竞品/风险/MVP）？ 不预置。首次 distill 让 LLM 自然组织出结构
Q3 跨点子聊天 / 引用其他点子？ 否。每场对话只属于一个点子
Q4 标签 / 分类 / 收藏？ MVP 不做。列表按 updated_ts 排序就够
Q5 移动 / 桌面入口位置？ 跟 agent_lord 一致：openFrom: "projects"，桌面 section: "projects"，label "点子库"
Q6 文档长度上限？ 软上限 50k 字符（distill 时校验，超了拒绝写入并报错）
Q7 对话里 LLM 能不能用工具（读文件/搜网）？ 否。MVP 纯文本对话，跟壳子裸聊一致（--permission-mode bypassPermissions 但不主动调工具）
Q8 删点子时连带删对话和历史 vs 软删除？ 硬删除 + 删除前在 server 端 log 路径（便于事后从备份找回）
